### 1.1 Authentication Module (`Core.Authentication`)

Defines a thread-safe authentication provider using the Microsoft Authentication Library (MSAL.NET) to manage multi-tenant token acquisition and localized in-memory token caching.

#### Detailed Class Specifications: `MsalAuthenticationProvider.cs`

* **Token Caching Strategy**: Must maintain an internal, static, concurrent dictionary (`ConcurrentDictionary<string, IConfidentialClientApplication>`) keyed by Tenant ID and Client ID to prevent app instantiation overhead.
* **Auth Flow Support**:
* **ClientSecret**: Uses `ConfidentialClientApplicationBuilder`.
* **Interactive**: Uses `PublicClientApplicationBuilder` with `.WithRedirectUri("http://localhost")` for developer convenience.


* **Scope Mapping**: Appends `/.default` to the incoming environment URL (e.g., `https://org.crm.dynamics.com/.default`) to enforce RBAC permissions.

### 1.3 Resilience & Throttling Module (`Core.Resilience`)

A critical layer that intercepts Dataverse Service Protection API Limits (HTTP 429) and coordinates transient failure recovery across multi-threaded operations.

#### Class Logic: `ThrottlingHttpHandler.cs` (Inherits from `DelegatingHandler`)

1. **Intercept Requests**: Wraps the underlying `SendAsync` method in a localized retry loop (maximum 5 retries).
2. **Evaluate Status**: If the response returns `HttpStatusCode.TooManyRequests` (429):
* Read the `Retry-After` header string. If empty, default to an exponential backoff baseline formula: $Delay = 2^{Attempt} \times 1000 \text{ milliseconds}$.
* Instantiate a centralized, cross-thread `SemaphoreSlim(1, 1)` to pause all other running parallel processing workers assigned to that target environment.
* Execute `Task.Delay(calculatedDelay)` before releasing the semaphore and executing the transaction retry block.


3. **Transient Fault Mapping**: Intercepts server-side HTTP codes `502 (Bad Gateway)`, `503 (Service Unavailable)`, and `504 (Gateway Timeout)` to process standard exponential retries.

### 2.2 Granular Phase Requirements & API Specifications

#### Phase 1: Local In-Memory Manifest Analysis

The utility must unpack the target solution package zip file entirely within memory streams (`System.IO.Compression.ZipArchive`) to enforce strict security barriers and ensure low file-system dependencies.

* **File Resource `solution.xml**`:
* Extract the unique solution identifier (`/ImportExportXml/SolutionManifest/UniqueName`).
* Extract target platform version bounds (`/ImportExportXml/@Version`).
* Parse the `<MissingDependencies>` element array. Map each item to a local structural model capturing `Required` components versus `Dependent` system hooks.


* **File Resource `customizations.xml**`:
* Scan for `<Entity>` elements. Build a thread-safe map of physical table schemas included in the package, noting whether the attribute `IncludeAllComponents` evaluates to `true`.



#### Phase 2: Async Solution Staging API Execution

Leverage the native Dataverse Staging framework to let the target ecosystem pre-evaluate structural import conflicts without modifying data layers.

* **API Action Target**: `StageSolution`
* **Web API Request Pattern**:
```http
POST [Organization URI]/api/data/v9.2/StageSolution HTTP/1.1
Content-Type: application/json

{
  "SolutionPackage": "[Base64EncodedZipBinaryString]"
}

```


* **Processing Routine**:
* Because large solutions take significant processing time, catch the returned processing transaction token.
* Poll the target platform environment by dispatching iterative GET requests tracking the execution processing state.
* Extract the raw response content block containing the structural platform verification data. Map the platform errors to the internal validation report.



#### Phase 3: Parallel Target Environment Delta Crawling

To catch issues that standard platform staging checks miss, the utility runs parallel asynchronous verification loops using the Core project's connection pool.

##### A. Table and Column Metadata Alignment

For each record parsed from `customizations.xml`, verify schema parity in the target target environment.

* **Web API Endpoint**:
```http
GET /api/data/v9.2/EntityDefinitions(LogicalName='{table_logical_name}')?$select=LogicalName,IsCustomEntity,VersionNumber&$expand=Attributes($select=LogicalName,AttributeType)

```


* **Validation Rules Engine**:
* **Rule A1**: If the table endpoint returns a `404 Not Found` status and the schema is flagged as out-of-the-box (OOB), mark it as a **Red Blocker** (Target environment missing platform prerequisites or updates).
* **Rule A2**: If the table exists, compare the properties of all custom components. If an attribute length, type, or option-set index configuration differs from the source file metadata, mark it as a **Red Blocker** (Metadata Conflict).



##### B. Target Unmanaged Layer Detection

Check components included in the solution for active unmanaged layers in the target environment that could block updates.

* **Web API Endpoint**:
```http
GET /api/data/v9.2/msdyn_componentlayers?$filter=msdyn_componentid eq '{component_guid_from_manifest}' and msdyn_solutionname eq 'Active'&$select=msdyn_componentname,msdyn_componenttype

```


* **Validation Rules Engine**:
* **Rule B1**: If the query returns a populated JSON record array, an unmanaged layer is active on the target component. Mark it as a **Yellow Warning**. The report must explicitly output: *"Unmanaged layer detected on target component {name} ({guid}). Managed solution updates will be ignored at runtime unless this layer is cleared."*

### 3.2 Detailed Operational Requirements & API Specifications

#### Phase 1: Bloat Matrix Diagnostics

The component scanner parses `customizations.xml` to flag solution components that pull in unintended configurations.

* **Manifest Evaluation Rules**:
* Locate target `<RootComponent type="1" ...>` nodes (where `type="1"` maps to the Dataverse Entity metadata wrapper).
* Evaluate the solution composition configuration profile. If an out-of-the-box system component definition (e.g., Account, Contact, Lead, Opportunity) lacks precise component tracking attributes and includes the entire asset array, flag it for pruning.



#### Phase 2: Decomposition and Component Optimization

The utility uses a clean recalculation pipeline to determine the minimum required assets, switching the reference type from broad inclusion to targeted tracking.

* **Calculated Dependencies Target Web API Endpoint**:
```http
POST /api/data/v9.2/AddSolutionComponent HTTP/1.1
Content-Type: application/json

{
  "ComponentId": "{specific_attribute_or_form_guid}",
  "ComponentType": 2, // e.g., Attribute
  "SolutionUniqueName": "{target_solution_name}",
  "AddRequiredComponents": false,
  "DoNotIncludeSubcomponents": true
}

```


* **Refactoring Automation Sequence**:
1. The tool tracks only the specific fields, forms, or views that have unmanaged changes or are explicitly called by custom logic.
2. It uses the Dataverse Web API action `RemoveSolutionComponent` to drop the broad entity root component node from the solution configuration schema tree.
3. It updates the configuration records, passing `DoNotIncludeSubcomponents = true` to the entity registration API.
4. It runs a parallel async batch loop using the `AddSolutionComponent` API action to explicitly register only the approved, modified components.



#### Phase 3: Differential Manifest Packaging

The utility outputs a optimized solution package along with a clear structural report of the changes.

* **Output Requirements**:
* **Optimized Solution Export Archive**: Generates a clean, smaller `.zip` file containing only the required component nodes.
